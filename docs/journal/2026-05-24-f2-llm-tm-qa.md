## Correctif UI — Proportions ResizablePanelGroup (post-F2)

**Problème** : Les 3 panneaux horizontaux (FileTree, SegmentGrid, SidePanel) étaient mal proportionnés — FileTree et SidePanel quasiment invisibles, SegmentGrid prenait ~95% de la largeur.

**Cause racine** : `react-resizable-panels` v4 (installé `^4.11.1`) a changé la sémantique des `defaultSize` numériques : **un nombre est maintenant traité comme des pixels** (pas des %), via la fonction interne `ht(n)` qui retourne `[n, "px"]` pour `typeof n === "number"`. Ainsi `defaultSize={20}` = 20px, soit ~2% d'un groupe de 913px, au lieu de 20%. Les panneaux latéraux étaient dimensionnés à ~20px et ~25px, le centre récupérait le reste (~95%).

**Fix** : Toutes les props de taille du `ResizablePanelGroup` horizontal passées en strings avec suffixe `%` :
- FileTree : `defaultSize="20%"` `minSize="15%"` `maxSize="35%"`
- SegmentGrid : `defaultSize="55%"` `minSize="40%"`
- SidePanel : `defaultSize="25%"` `minSize="20%"` `maxSize="40%"`

Ajout de `collapsible={false}` sur les 3 panneaux et `withHandle={true}` sur les 2 poignées horizontales.

**À noter** : Les panneaux VERTICAUX internes (TM/QA) continuaient à fonctionner avec des nombres car la lib calcule leur groupSize différemment à l'initialisation. La règle générale pour v4 : **toujours utiliser des strings `"N%"` pour les props de taille** afin d'être explicite.

**Vérification** : `pnpm typecheck` ✅ 0 erreurs. Screenshot `docs/screenshots/f2-ui-panels-fix.jpeg` confirme les 3 panneaux correctement proportionnés.

## Correctif FileTree — noms de fichiers manquants

**Problème** : Le FileTree affichait uniquement les icônes, sans les noms de fichiers. Le projet était bien chargé (~25 fichiers), mais chaque ligne n'affichait qu'une icône vide.

**Cause racine** : Tauri/serde **ne convertit PAS automatiquement** les noms de champs Rust en camelCase. Sans `#[serde(rename_all = "camelCase")]`, le champ Rust `file_name` est sérialisé en JSON comme `file_name` (snake_case). Le TypeScript accédait à `file.fileName` → `undefined`. Cette erreur affectait silencieusement TOUS les champs composés : `fileType`, `projectId`, `sourceText`, `targetText`, `qaScore`, etc.

La mention dans CONTEXT.md "Tauri sérialise en camelCase automatiquement" était **incorrecte**. Corrigée.

**Fix** : Ajout de `#[serde(rename_all = "camelCase")]` sur 7 structs IPC :
- `commands/project.rs` : `Project`, `SourceFile`, `Segment`, `PaginatedSegments`, `ProviderConfig`, `QaReport`
- `core/tm.rs` : `TmEntry`

`sqlx::FromRow` n'est pas affecté par les attributs serde — le mapping DB reste correct.

**À noter** : Lors de chaque recompilation Rust via `tauri dev`, le store Zustand (en mémoire) est réinitialisé — il faut rouvrir le projet.

**Vérification** : `cargo test` ✅ 83/83, `cargo clippy -D warnings` ✅, `pnpm typecheck` ✅ 0 erreurs. CONTEXT.md mis à jour avec la règle correcte.

## Correctif — `translate_segments` ne faisait rien (post-F2)

**Problème** : Cliquer "Traduire" ne produisait aucun effet.

**Cause racine** : La chaîne d'appel transmettait toujours `ids: []` (tableau vide) :
1. `LlmConfigModal.handleStart` → `onStart([])`
2. `startTranslation([])` → `invoke("translate_segments", { ids: [] })`
3. Rust : `if ids.is_empty() { return Ok(()); }` → sortie immédiate silencieuse

Le `file_id` n'était jamais transmis au backend malgré le commentaire "The store/command handles ID lookup".

**Fix** :
- `translate_segments` accepte `file_id: Option<String>` : quand `ids` est vide et `file_id` fourni, fetch tous les segments `untranslated` ou `target_text = ''` du fichier
- `startTranslation(segmentIds, fileId?)` passe le `fileId` au `invoke`
- `handleStart` transmet `activeFileId` depuis le store

**Résultat vérifié** : Actors.json — 6 segments traduits (ラディア → Radiya, 見張り兵 → Sentry, etc.)

**Vérification** : `cargo test` ✅ 83/83, `cargo clippy -D warnings` ✅, `pnpm typecheck` ✅ 0 erreurs.

## Correctif — SegmentGrid ne se rafraîchit pas après traduction LLM (post-F2)

**Problème** : Après une traduction batch, les segments affichaient toujours "Non traduit" jusqu'à ce qu'on quitte le fichier et y revienne.

**Cause racine** : `SegmentGrid` fetche les segments une seule fois au changement de `activeFileId` (`useEffect` sur `[activeProjectId, activeFileId]`). La pipeline LLM tourne dans un `tokio::spawn` côté Rust et met à jour la DB sans que le frontend en soit notifié. L'event `h2s://llm/completed` était écouté uniquement dans `llm.ts` pour mettre à jour `isTranslating` — mais personne ne déclenchait un re-fetch des données.

**Fix** dans `SegmentGrid.tsx` :
- Logique de fetch extraite dans `loadSegments` (`useCallback` stable)
- Refs `activeProjectIdRef` / `activeFileIdRef` pour capturer les IDs courants sans re-créer le listener
- Second `useEffect` qui s'abonne à `h2s://llm/completed` (une seule fois, cleanup propre) et appelle `loadSegments` — les segments traduits apparaissent immédiatement sans quitter le fichier

**Vérification** : `pnpm typecheck` ✅ 0 erreurs.
