## Key technical findings

### Two-file format (critical discovery)
Wolf RPG Database is stored as two separate files:
- `.project` — schema (type names, field names, field indexInfo)
- `.dat` — data values (int arrays + string arrays per entry)

Both must be parsed together. `.project` provides the schema; `.dat` contains values without any field labels.

### Magic bytes
```
SJIS: 57 00 00 4F 4C 00 46 4D 00
UTF8: 57 00 00 4F 4C 55 46 4D 00
```
Byte 5 distinguishes encoding: `0x00` = Shift-JIS, `0x55` = UTF-8.

### Field type detection (from WolfTL `Field::IsString`)
- `indexInfo >= 0x07D0` → string field
- `indexInfo >= 0x03E8` → valid/in-use field
- Data layout: all int values come first, then all string values (not interleaved)

### String format (ReadString)
`u32_le size` (includes null terminator) + `size` raw bytes. Size=0 is an error (corrupted file).

### LZ4 compression
Version byte `0xC4` = LZ4-compressed. Not supported in F4-03; returns `Unsupported` error.

### CommonEvents — Cas C (stub)
After reading `CommonEvents.hpp` from WolfTL, the format is far too complex to implement safely without a real test fixture:
- Per-event: `0x8E` indicator byte + 7 unknown bytes + command list
- Then 100 fixed strings per event (0x8F section)
- Then 0x91 and 0x92 sections of unknown length

Any synthetic test would be guessing at the format. Chose to stub with `Ok(vec![])` and a `TODO(F4-05)` comment. CommonEvents text is a lower priority than database and map text.

## String extraction heuristic

Two-pass filter on string fields:
1. **Known translatable field names**: `name`, `名前`, `description`, `説明`, `note`, `備考`, `message`, `text`
2. **Contains Japanese**: hiragana `\u{3041}–\u{3096}`, katakana `\u{30A0}–\u{30FF}`, CJK `\u{4E00}–\u{9FFF}`

Empty strings and strings matching `\[` (placeholders) are skipped.

Key format: `Database/{db_name}/{type_idx}/{entry_idx}/{field_name}`

## Dead code strategy

Several dat_parser fields (`DatEntry::name`, `int_values`, `DatType::name`, `DatFile::is_utf8`) are written by the parser but not read by the extractor. They are needed by the injector (F4-04). Used targeted `#[allow(dead_code)]` with "F4-04 injector" comments rather than removing the fields.
