## Ce qui a été fait

### Step 6 — GuessKeyV6 (commit dd991f7)

Fichier : `src-tauri/src/engines/wolf/decryptor.rs`

- `guess_key_v6(raw_header: &[u8]) -> Option<[u8; 12]>` — `pub fn`
- Principe : les 4 octets hauts des champs i64 sont nuls en clair → XOR révèle les bytes de clé
  - `key[0..4]` = `file[0x0C..0x10]` (high bytes de base_offset)
  - `key[4..8]` = `file[0x1C..0x20]` (high bytes de file_table)
  - `key[8..12]` = `file[0x14..0x18]` (high bytes de index_offset)
- Deux validations croisées :
  - `file[0x0C..0x10] == file[0x24..0x28]` (dir_table high = base_offset high, même key_pos)
  - `file[0x14..0x18] == file[0x2C..0x30]` (post-header zeros, même key_pos = 8..11)
- Validation additionnelle : version ∈ {5, 6, 8} + index_size décrypté < 0x1000000
- 2 tests : `test_guess_key_v6_synthetic`, `test_guess_key_v6_random_data`

**Point clé découvert** : les bytes `file[0x2C..0x30]` (post-header zeros) sont chiffrés avec
key_conv — l'encryption s'étend JUSQU'À 0x30 inclus pour la validation, même si le corps
formel de l'en-tête s'arrête à 0x2C. `make_v6_header_for_guess` chiffre donc 44 bytes
(file[0x04..0x30]) pour que la validation 2 soit satisfaite dans le test.

## Points ouverts documentés

### LZSS — Absent dans les archives synthétiques

Aucune archive synthétique du test ne génère d'entrées LZSS (`packed_size > 0` ≠ `unpacked_size`).
Toutes les archives de test utilisent `packed_size = -1` (non compressé).

**Conclusion pour F4** : `UnsupportedCompression` est retourné si rencontré, mais non testé
activement. Les jeux de test réels (月咲流ホノカ, Densyanai Inko) devront être vérifiés
lors de l'intégration F4-05 pour confirmer que LZSS n'est pas utilisé.

### GuessKeyV6 — Validation sur le test synthétique

La validation fonctionne correctement. Le point crucial est que `file[0x2C..0x30]` (post-header
zeros) **sont chiffrés** avec key_conv (l'encryption s'étend au-delà du corps formel à 0x2C).
`make_v6_header_for_guess` chiffre 44 bytes = file[0x04..0x30] pour simuler ce comportement.

### Noms de fichiers — Lecture depuis la TOC

Les noms sont lus depuis `toc_data` **après** que `parse_index` l'a déchiffré en place.
`name_offset` est un offset absolu dans la TOC (depuis l'octet 0). Le null-terminator
est localisé par `position(|&b| b == 0)`.
